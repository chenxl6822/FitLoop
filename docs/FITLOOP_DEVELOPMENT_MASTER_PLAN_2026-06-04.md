## 1. 项目目标

FitLoop 当前是课程设计项目，不应按商业级复杂系统无限扩张。下一阶段的目标是让核心功能完整、稳定、可演示、逻辑闭环：

1. 账号能注册、登录、找回密码。
2. 运动目标能真实保存、展示、编辑或删除。
3. 运动打卡能根据运动类型给出合理方式。
4. GPS 打卡即使没有地图 SDK，也要能采集轨迹并给出可见反馈。
5. “我的”模块头像、提醒设置、反馈入口可用。
6. 健康统计图表不重叠、不伪造趋势。
7. 管理端至少有最小可演示版本。

## 3. 计划优化原则

### 3.1 功能取舍

优先做能被老师或测试者直接看到的闭环功能：

- 目标：创建后能保存、展示、删除、重新登录仍存在。
- 我的：头像能显示，提醒能保存，反馈能提交。
- 统计：图表显示可信，不出现错位或假趋势。
- 打卡：用户选什么运动，就给合理的打卡方式。

暂缓高成本但非课程核心的功能：

- 真实短信服务。
- 完整地图 SDK。
- 对象存储 COS。
- 第三方登录。
- iOS 发布。
- 商业级权限体系。

### 3.2 外部服务策略

没有第三方服务时，不写假功能：

- 没有短信服务：手机验证码入口隐藏或提示“暂未开放”，用 `FITLOOP_SMS_ENABLED=false` 控制。
- 没有地图 SDK：先显示定位状态、经纬度、里程、轨迹点列表，预留地图接入说明。
- 没有推送服务：先保存提醒配置，本地通知可用则开启，本地通知不可用时给出说明。
- 没有对象存储：继续使用服务器 uploads 目录，但确保 Docker volume 和 Nginx 可访问。

## 4. 手机验证码计划

### 当前结论

项目已有验证码生成、存储、过期、校验和防刷逻辑，但没有真实短信服务商配置。课程设计阶段不应把 debug 验证码当正式功能。

### 优化方案

1. 增加或文档化短信开关：`FITLOOP_SMS_ENABLED=false`。
2. 无短信配置时：
   - 前端隐藏“手机验证码登录/注册/找回”入口，或显示“手机验证码暂未开放，请使用邮箱”。
   - 邮箱验证码不受影响。
3. 开发/测试环境：
   - 可允许 `FITLOOP_OTP_DEBUG_RETURN=true` 返回 debug code。
   - UI 必须写“开发环境验证码”或“内测验证码”，不能写“短信已发送”。
4. 未来接短信服务：
   - 保留 `VerificationCodeSender` 抽象。
   - 替换 `PhoneVerificationCodeSender`。
   - 增加短信签名、模板 ID、SecretId、SecretKey 等配置。

## 6. 第一优先级任务

### 6.1 运动目标模块

必须确认：

- 创建目标真实入库。
- 用户 ID 绑定正确。
- 默认目标和用户自定义目标不混乱。
- 列表展示目标名称、类型、值、单位、开始时间、结束时间、完成进度。
- 删除目标能删除用户创建的目标。
- 增加编辑目标能力，至少可修改目标值、周期、指标。

### 6.2 “我的”模块

头像：

- `.jpg` / `.jpeg` / `.png` 上传成功。
- 合法 `.jpg` 不再提示“请上传图片文件”。
- 上传后立即刷新。
- 重启 App 或重新登录后仍显示。
- `/uploads/avatars/**` 公网可访问。

提醒：

- 提醒开关能保存。
- 提醒时间能修改。
- 重新进入页面后配置仍存在。
- 没有真实推送时，文档说明使用本地通知或预留推送服务。

### 6.3 意见反馈

课程演示价值高，建议进入第一优先级：

- 前端有入口。
- 后端有 `POST /api/feedback`。
- 数据库保存用户 ID、类型、内容、联系方式、提交时间、状态。
- 管理端可查看和处理。

### 6.4 健康统计图表

必须修复：

- 顶部数字不重叠。
- 横坐标不重复。
- 日期升序。
- 不强行补 0 点。
- 0 条数据显示空状态。
- 1 条数据显示单点或摘要。
- 2 条以上再绘制趋势。

### 6.5 手机验证码入口

无短信配置时不能出现不可用入口。邮箱验证码继续可用。

## 7. 第二优先级任务

### 7.1 智能打卡方式

户外运动：

- 跑步、健走、骑行、徒步。
- 显示 GPS、计时、手动。

室内运动：

- 跳绳、力量训练、瑜伽、拉伸、健身操、室内单车。
- 显示计时、次数、手动、热量估算。
- 不默认推荐 GPS。

### 7.2 GPS 可用降级

无地图 SDK 时，也必须显示：

- 定位状态。
- 当前经纬度。
- 已上传轨迹点数。
- 里程。
- 用时。
- 速度或配速。
- 暂停/继续/结束。

地图 SDK 后续再接入，不能用假地图。

### 7.3 拍照打卡取舍

如果拍照打卡稳定：

- 相机/相册可用。
- 上传可用。
- 记录保存 photoUrl。
- 记录详情能查看图片。

如果不稳定：

- 暂时隐藏入口。
- 用计时打卡或运动数据打卡替代。
- 文档记录恢复条件。

## 8. 第三优先级：后台管理 MVP

后台不是一开始就做大而全，而是先做课程演示最小版本。

最小版本：

1. 管理员登录或 admin key 登录。
2. 用户列表：手机号、邮箱、状态、注册时间、目标数、记录数。
3. 用户详情：资料、目标、运动记录、健康数据。
4. 反馈管理：列表、筛选、标记已处理、备注。
5. 数据看板：总用户、今日新增、总运动记录、今日打卡、未处理反馈。

权限要求：

- 后台接口统一 `/api/admin/**`。
- 普通用户 JWT 不能访问。
- 管理员密钥或管理员账号不能硬编码在前端。

## 9. 测试计划

每个 Sprint 至少执行：

- 后端：`mvn test`
- 移动端：`flutter analyze`
- 移动端：`flutter test`
- 真机：注册、登录、目标、打卡、头像、提醒、反馈、统计
- 部署：`/actuator/health`、APK 下载、uploads 资源访问

重点测试数据：

- 手机号账号和邮箱账号不冲突。
- 目标数据重新登录后仍存在。
- 头像 URL 在公网可访问。
- 提醒配置重新进入页面后仍存在。
- 图表数据来自真实运动记录。

下一阶段的核心不是“功能越多越好”，而是把课程设计最容易被检查的问题修干净：目标、头像、提醒、反馈、图表、短信入口、打卡方式和后台管理 MVP。所有外部服务能力都用配置开关和预留接口处理，避免假功能和半成品入口。
