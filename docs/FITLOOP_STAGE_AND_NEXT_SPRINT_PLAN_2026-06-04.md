# FitLoop 当前阶段与下一阶段任务规划

> 日期：2026-06-04  
> 当前目标：课程设计可演示闭环。  
> 本次约束：只修改三份规划文档，不修改代码。

## 1. 当前阶段判断

项目已进入“核心功能补全与演示稳定化”阶段。代码中已经出现许多能力雏形，例如验证码服务、目标创建/删除、GPS 轨迹上传、头像上传、提醒保存、统计图表、申诉审核等。但课程设计验收不会只看“有没有入口”，更会看：

- 数据是否真实入库。
- 重新登录后是否仍存在。
- 删除和编辑是否完整。
- 页面是否有半成品按钮。
- 错误提示是否清楚。
- 外部服务缺失时是否诚实降级。

因此下一阶段规划从原来的“账号与目标稳定化”扩展为“第一优先级问题集中修复”。

## 2. 当前开发计划检查结论

原计划优点：

- 已识别验证码、目标、头像、GPS、部署等关键问题。
- 已强调无短信服务不能假装短信已发送。
- 已把 `.jpg` 头像识别纳入要求。

需要优化的地方：

- 目标模块不应只做删除，还要确认保存、展示、编辑、默认目标区分和字段映射。
- “我的”模块不只头像，还包括提醒时间和开关保存。
- 意见反馈应提前到第一优先级，因为它和后台管理 MVP 强相关。
- 图表问题应提前处理，避免演示时出现明显 UI 错误。
- 手机验证码策略应从“内测回显”进一步收敛为 feature flag：无短信服务时隐藏或提示暂未开放。
- 后台管理不能只放在远期，应至少规划 MVP。

## 3. 下一阶段总优先级

### 第一优先级：必须修复

1. 运动目标保存、展示、编辑、删除。
2. “我的”模块头像显示和提醒设置保存。
3. 意见反馈真实入库。
4. 健康统计图表显示异常。
5. 手机验证码不可用时隐藏入口，并保留后续短信能力。

### 第二优先级：演示建议修复

1. 运动打卡根据运动类型智能显示打卡方式。
2. GPS 打卡无地图 SDK 时提供可用降级方案。
3. 拍照打卡隐藏或替换为稳定打卡方式。

### 第三优先级：后台管理

1. 管理员登录。
2. 用户列表。
3. 用户详情。
4. 反馈查看和处理。
5. 基础统计看板。

## 4. Sprint 1：核心闭环修复

目标：让课程设计最容易被点到的功能不再是半成品。

建议工时：8-12 小时。

### 任务 1.1：运动目标模块

检查范围：

- 创建目标接口。
- 数据库 `sport_target` 或相关表。
- 用户 ID 绑定。
- 默认目标和用户目标区分。
- 当前目标列表查询。
- 删除接口。
- 编辑接口。
- 前端状态刷新。
- 时间、指标、目标值、单位字段映射。

优化后的验收标准：

- 创建目标后列表立即显示。
- 重新进入页面后仍显示。
- 展示目标名称、类型、值、单位、开始时间、结束时间、进度。
- 删除时能看到用户创建的目标，并能删除。
- 编辑目标后数据库和页面同步。
- 默认目标和用户目标不混乱。
- 不允许前端假显示。

### 任务 1.2：“我的”模块

头像：

- 支持 `.jpg` / `.jpeg` / `.png`。
- 合法 `.jpg` 不误报“请上传图片文件”。
- 头像上传后立即显示。
- 重新进入“我的”页面后仍显示。
- 服务器部署时 `/uploads/avatars/**` 可访问。

提醒：

- 提醒设置表、查询接口、保存接口可用。
- 开启/关闭字段可保存。
- 提醒时间字段可修改。
- 保存后重新读取。
- 重启 App 后仍保留。

验收标准：

- 头像上传、显示、持久化完整可用。
- 提醒时间可以修改。
- 提醒开关可以保存。
- 页面不再只是静态摆设。

### 任务 1.3：意见反馈真实入库

建议实现范围：

- 新增反馈表或实体。
- 字段：用户 ID、类型、内容、联系方式、提交时间、状态、管理员备注。
- 前端“我的/设置”增加反馈入口。
- 后端 `POST /api/feedback`。
- 后续后台通过 `/api/admin/feedback` 查看。

验收标准：

- 用户能提交反馈。
- 反馈真实保存。
- 提交成功有明确提示。
- 后台规划中能查看反馈。

### 任务 1.4：健康统计图表修复

检查范围：

- `mobile/lib/stats_charts.dart`。
- 运动历史接口数据排序。
- 横坐标标签生成。
- 单点和空数据处理。
- 是否为了画线补 0。

验收标准：

- 顶部统计数字不重叠。
- 横坐标不重复。
- 日期升序。
- 曲线不会无原因先下降。
- 0 条数据显示空状态。
- 1 条数据显示单点或摘要。
- 2 条以上正常绘制趋势。

### 任务 1.5：手机验证码入口策略

建议策略：

- 增加或明确 `FITLOOP_SMS_ENABLED=false`。
- 无短信服务时隐藏手机验证码入口，或提示“手机验证码暂未开放，请使用邮箱”。
- 邮箱验证码继续可用。
- debug 验证码只允许开发/测试环境。

验收标准：

- 没有短信配置时用户看不到不可用入口，或看到明确暂未开放。
- 邮箱验证码不受影响。
- 后续配置短信服务后可快速启用。

## 5. Sprint 2：运动打卡体验优化

目标：运动类型和打卡方式匹配，不再让用户进入不合理或不可用流程。

建议工时：6-8 小时。

### 任务 2.1：运动类型和打卡方式映射

户外运动：

- 跑步、健走、骑行、徒步。
- 可显示 GPS、计时、手动。

室内运动：

- 跳绳、力量训练、瑜伽、拉伸、健身操、室内单车。
- 可显示计时、次数、手动、热量估算。

验收标准：

- 跳绳不默认推荐 GPS。
- 跑步、骑行可看到 GPS。
- 不同运动类型输入字段合理。
- 保存后的记录字段准确。

### 任务 2.2：GPS 打卡降级方案

在没有地图 SDK 或 API Key 前，不做假地图。

先实现或验收：

- 定位权限检查。
- 当前定位状态。
- 当前经纬度。
- 已记录轨迹点数。
- 里程。
- 用时。
- 速度或配速。
- 暂停/继续/结束按钮。
- 后端保存轨迹点。

后续地图接入：

- 高德、腾讯或 Google Maps 任选其一。
- Key 只能通过配置注入。
- 不写死密钥。

### 任务 2.3：拍照打卡取舍

如果保留：

- 拍照/相册选择。
- 上传图片。
- 保存记录。
- 查看图片。
- 权限提示。

如果不稳定：

- 暂时隐藏入口。
- 替换为计时打卡或运动数据打卡。
- 文档记录恢复条件。

## 6. Sprint 3：后台管理 MVP

目标：用最小成本支持课程设计演示。

建议工时：8-12 小时。

### 必做能力

1. 管理员登录或 admin key 登录。
2. `/api/admin/**` 接口保护。
3. 用户列表：
   - 用户 ID。
   - 手机号/邮箱。
   - 状态。
   - 注册时间。
   - 目标数量。
   - 运动记录数量。
   - 最近活跃时间。
4. 用户详情：
   - 基本资料。
   - 运动目标。
   - 运动记录。
   - 健康数据。
5. 反馈管理：
   - 反馈列表。
   - 类型筛选。
   - 状态筛选。
   - 标记已处理/未处理。
   - 管理员备注。
6. 基础统计：
   - 总用户数。
   - 今日新增用户。
   - 总运动记录数。
   - 今日打卡数。
   - 反馈总数。
   - 未处理反馈数。

### 实现形式建议

如果时间紧：

- 后端先提供 `/api/admin/**` JSON 接口。
- 前端可做一个简单 Web 管理页，或移动端隐藏 admin 页面。
- 不追求复杂 UI，优先数据真实和权限正确。

## 7. Sprint 4：部署与验收

目标：服务器演示时不翻车。

任务：

- 同步 README 和 STATUS。
- 检查 `.env`。
- 检查 `FITLOOP_SMS_ENABLED`、`FITLOOP_OTP_DEBUG_RETURN`。
- 检查 uploads volume。
- 检查 Nginx `/uploads/avatars/**` 和 `/uploads/photos/**`。
- 检查 APK 下载页。
- 跑完整真机冒烟。

验收：

- APK 可下载。
- API 可访问。
- 上传头像可公网访问。
- 数据库持久化。
- 反馈和目标重新登录后仍存在。

## 8. 推荐执行顺序

1. 目标模块保存/展示/编辑/删除。
2. 头像显示和提醒保存。
3. 意见反馈真实入库。
4. 图表修复。
5. 手机验证码入口 feature flag。
6. 智能打卡方式。
7. GPS 降级。
8. 拍照打卡取舍。
9. 后台管理 MVP。
10. 部署验收。

## 9. 输出要求

后续真正改代码时，每轮完成后要输出：

- 问题根因。
- 修改文件清单。
- 数据库是否需要迁移。
- 本地测试方式。
- Android 真机测试方式。
- 服务器部署验证方式。
- 未完成但已预留的能力。

## 10. 本阶段完成定义

本阶段不以“写了多少功能”为完成标准，而以“演示路径是否真实可用”为标准：

- 邮箱验证码不被破坏。
- 手机验证码无服务时不误导用户。
- 目标数据真实保存和可管理。
- 头像和提醒真实可用。
- 反馈真实入库。
- 图表可信。
- 打卡方式合理。
- 后台管理有 MVP 规划并可开始实现。
