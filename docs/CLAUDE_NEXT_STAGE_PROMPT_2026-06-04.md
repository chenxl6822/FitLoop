## 1. 你的任务定位

你负责修复 FitLoop 项目。FitLoop 当前处于课程设计阶段，目标不是商业级复杂系统，而是让核心功能完整、稳定、可演示、逻辑闭环。

3. 最后补测试和说明。

## 2. 总体要求

3. 涉及数据库字段、接口、模型、页面状态、缓存、权限时，保持前后端一致。
4. 缺少第三方服务配置时，不硬写假功能；使用 feature flag、配置项和 TODO 文档预留。
5. 所有新增或修改功能必须保证：
   - 页面正常显示。
   - 数据真实保存。
   - 刷新或重新登录后数据仍存在。
   - 删除、编辑、查询功能完整。
   - 错误提示清晰。
   - 不出现空白页、假成功、前端显示和后端数据不一致。

## 4. 第一优先级：必须修复

### 4.1 手机验证码功能

检查：

- 手机号注册。
- 手机号登录。
- 手机号忘记密码。
- 验证码生成、存储、过期、校验。
- 防刷机制。
- 短信服务配置。
- 前端手机号输入和验证码按钮。
- 手机号和邮箱账号体系是否冲突。

处理要求：

1. 如果有真实短信服务配置，让手机验证码完整可用。
2. 如果没有真实短信服务商配置，暂时隐藏手机验证码入口，或提示“手机验证码暂未开放，请使用邮箱验证码”。
3. 保留后续开发能力：
   - 后端保留 sender/provider 抽象。
   - 前端保留可恢复入口。
   - 配置预留 `FITLOOP_SMS_ENABLED=false`。
   - 文档记录如何重新开启。
4. 不允许前端把验证码回显当正式功能。
5. debug 验证码只能在开发/测试环境开启，生产必须关闭。

验收：

- 无短信配置时，用户看不到不可用入口，或看到明确暂未开放。
- 邮箱验证码不受影响。
- 配置短信服务后能快速启用。

### 4.2 运动目标模块

当前重点不只是“删除”，而是完整目标管理闭环。

检查：

- 创建目标接口。
- 目标保存逻辑。
- 数据库目标表。
- 用户 ID 绑定。
- 默认目标和用户自定义目标区分。
- 前端创建后状态刷新。
- 目标列表查询。
- 删除目标接口。
- 编辑目标接口。
- 时间、指标、目标值、单位字段映射。

修复要求：

- 创建目标真实入库。
- 列表显示所有当前用户目标。
- 删除能删除用户创建的目标。
- 增加编辑目标能力，至少能修改指标、目标值、周期。
- 前端显示目标名称、类型、目标值、单位、开始时间、结束时间、完成进度。

验收：

- 创建后立即显示。
- 重新进入页面仍显示。
- 删除选项能看到用户创建目标。
- 删除后数据库和页面同步。
- 默认目标和用户目标不混乱。
- 不允许前端假显示。

### 4.3 “我的”模块

头像：

- `.jpg`、`.jpeg`、`.png` 必须支持。
- 合法 `.jpg` 不应提示“请上传图片文件”。
- 头像上传后立即显示。
- 重新进入页面仍显示。
- 检查头像 URL 拼接、静态资源访问、uploads 目录映射、Nginx。

提醒：

- 可开启和关闭。
- 可修改时间。
- 可保存。
- 重新读取后仍存在。
- 若没有真实推送，至少保存提醒配置并说明后续如何接本地通知或推送。

验收：

- 头像上传、显示、持久化完整可用。
- 提醒时间和开关保存成功。
- 页面不再只是静态摆设。

### 4.4 意见反馈

新增或完善：

- 前端反馈入口。
- 后端反馈接口。
- 数据库反馈表。
- 字段：类型、内容、联系方式、提交时间、用户 ID、状态、管理员备注。
- 提交成功提示。
- 管理端可查看反馈。

验收：

- 用户可提交反馈。
- 反馈真实入库。
- 管理端能查看和处理。

### 4.5 健康统计图表

检查：

- 图表组件。
- 数据源。
- 数据排序。
- 横坐标生成。
- 纵坐标范围。
- 空数据处理。
- 单条数据处理。
- 是否错误补 0。

修复要求：

- 顶部统计数字不重叠。
- 横坐标不重复。
- 数据按日期升序。
- 不强行在开头补 0。
- 0 条数据显示空状态。
- 1 条数据显示单点或摘要。
- 2 条以上正常绘制趋势。
- 曲线基于真实运动记录。

验收：

- 图表不重叠。
- 横坐标不重复。
- 曲线不会无原因先下降。
- 空数据和少量数据展示合理。

## 5. 第二优先级：运动打卡体验

### 5.1 按运动类型展示打卡方式

户外运动：

- 跑步、健走、骑行、徒步。
- 显示 GPS、计时、手动。

室内运动：

- 跳绳、力量训练、瑜伽、拉伸、健身操、室内单车。
- 显示计时、次数、手动、热量估算。
- 不默认推荐 GPS。

验收：

- 跳绳不推荐 GPS。
- 跑步、骑行可看到 GPS。
- 不同运动类型输入字段合理。
- 保存后的记录字段准确。

### 5.2 GPS 打卡地图和轨迹

检查：

- 是否接入地图 SDK。
- 是否申请定位权限。
- 是否获取实时经纬度。
- 是否记录轨迹点。
- 是否有运动开始、暂停、结束逻辑。
- 是否保存轨迹到后端。
- 是否能回看轨迹。

处理：

- 如果没有地图 SDK，不做假地图。
- 先实现降级：定位状态、经纬度、轨迹点数、里程、用时、速度/配速、暂停/继续/结束。
- 地图 SDK 作为后续可选，Key 通过配置注入。

验收：

- GPS 打卡不再是空白或无反馈。
- 能看到定位状态。
- 能记录轨迹。
- 结束后记录保存距离、时间、轨迹。

### 5.3 拍照打卡取舍

检查：

- 相机调用。
- 图片上传。
- 图片展示。
- 记录保存。
- 权限问题。
- 上传后查看。

处理：

- 如果稳定，保留并补齐闭环。
- 如果不稳定，暂时隐藏入口。
- 可替换为手动、计时、运动数据打卡。

验收：

- 不出现不可用拍照入口。
- 替代打卡模式能真实创建记录。

3. 用户详情。
4. 反馈管理。
5. 基础统计。

最小验收：

- 管理员能查看用户列表和运动数据。
- 管理员能查看反馈。
- 管理员能标记反馈处理状态。
- 普通用户不能访问。
